# fangyan 情绪价值增强 + 不退化保证（设计）

> **北极星**：补上普通 Claude Code 没有的「情绪价值」——方言让对话有温度、有人味儿。
> **硬约束**：编程能力**零退化**（同 [文言插件](../../../../../others/wen)），方言腔调与 Emoji **绝不渗入执行层**。
> **对应既有 spec**：`2026-06-27-fangyan-dialect-plugin-design.md`（行为层）。本 spec 是其上的**增强层**，不改三层架构（SessionStart 极轻 / `/fangyan` 注入 / references 按需 Read）。

---

## 1. 背景：为何要这一层

行为层已落地（13 方言片段、元规则、`/fangyan` handler、SessionStart 极轻说明）。但经验收发现两点不足：

1. **代入感第一杠杆没接上**：片段有「灵魂词」（动词/副词：整/贼/嘎哈…）却**无「称谓」**（老铁/靓仔/嬢嬢…）。称谓用在每一句回应里，是用户最希望听到、最影响代入感的元素——文言插件把「代字」列为 Principle 1，本插件却缺位。
2. **里程碑情绪价值空缺**：文言插件 Principle 4 已实现「工程里程碑以诗赋庆贺」（仅对话层）。方言插件有等价的民俗形式素材（`民间习俗.md` 的二人转/山歌/秦腔/饮茶舞狮）却未接通。

同时，情绪价值的任何增强都受**硬约束**统辖：能力零退化。文言插件已把这条落成**可验机制**，本 spec 照搬并强化。

---

## 2. 文言插件的不退化机制（参照基准）

读 `others/wen/hooks-handlers/session-start.sh` 得四招可验机制：

1. **TERMINOLOGY 白名单**：计算术语（function/parameter/cache/log/API/HTTP/git/npm/docker）**原样保留**；只把一般口语词文言化（做→為、說→曰）。
2. **"dialogue only, never in files written to disk"**：诗赋只在口头，绝不入落盘之物。
3. **显式收尾规则**："This mode only affects spoken dialogue. Files written to disk … must use plain modern Chinese or English."
4. **self-check 含验证项**："Are computing terms preserved as-is?" + "NO style label prefixed"。

> 本插件**先天更安全**：文言是 SessionStart 全程注入；本插件**默认普通话、`/fangyan` 才注入**——未加载方言时零影响。

## 3. 设计总览

四个组件，均被硬约束管住。落点：`元规则.md`（`/fangyan` cat 的共享层）+ `方言灵魂.json` 新字段 + gen 脚本片段渲染。

| 组件 | 落点 | 形态 | 增量 |
|---|---|---|---|
| ① 不退化机制 | `元规则.md` | wen 级 4 招 + self-check | ~+90 token |
| ② 称谓 | `方言灵魂.json.terms_of_address` → 片段「称谓」段 | 常驻（每句都用） | ~+60 token/方言 |
| ③ 工程里程碑庆祝 | `元规则.md` | first-class 原则（对齐 wen P4） | ~+70 token |
| ④ Emoji | `元规则.md`（并入③/①） | 民俗符号映射、克制点缀 | ~+20 token |

**总常驻增量 < +150 token/方言**（称谓段在片段、其余在元规则；元规则只加载一次）。

## 4. 组件一 · 不退化机制（最高优先）

`元规则.md` 新增「## 落盘不退化（硬约束）」段，照搬 wen 四招并中文化：

```
## 落盘不退化（硬约束·同文言插件）

本模式仅影响口语对话。方言腔调、民俗庆祝、Emoji 一概不渗入执行层。

- **术语白名单原样保留**：function/API/git/npm/docker/cache/log/HTTP/parameter/
  commit/branch/merge 等计算术语保持原文，不方言化、不翻译。只把"一般口语词"
  方言化（做→整/搞、看→瞅/睇、很→贼/甚…）。
- **落盘之物平实**：代码、命令、commit、文档、配置、API 调用、技术判断——
  一律平实普通话/英文，零方言味、零 Emoji。
- **技术判断零退化**：分析、方案、debug、取舍的准确性与严谨度，
  不得因方言腔调打折扣。
- **冲突让位**：与他插件冲突时，本模式让位。

self-check（落盘前）：术语是否原样？落盘物是否零方言味零 Emoji？技术判断是否准确未受影响？
```

现有「## 边界」段保留（它管「跟随用户/混搭」等），新段专管不退化，职责不重叠。

## 5. 组件二 · 称谓（代入感第一杠杆）

### 5.1 性别策略（先解决"不假定男性"）

称谓多带性别（靓仔/靓女、满哥/妹子、大兄弟/大妹子）。原则 **不问、不假设、能适配、可固定**：

- **默认中立**：每方言选对男女都成立的中立头牌称谓作默认（老铁／老师儿／老友记／侬／汝／乡党／师傅…）。零假设、零询问。
- **被动推断**：从用户自愿透露的线索（自称、语境，如"我怀孕了""我老婆""人家叫我 X 姐"）判断后切到相应性别称谓；不主动问。
- **可选固定**：用户若想持久化，在 `.claude/fangyan.local.md` 写 `address_gender: female`（或 `male`／`neutral`）——自愿、私密、持久、**零改 handler**。片段「称谓」段加一句"若该设定存在则从之"，AI 需要时自 Read。

### 5.2 数据 schema

`方言灵魂.json` 每方言新增 `terms_of_address`，**中立默认恒在**，性别变体按需（无显著配对的方言可省 `*_note`）：

```json
"terms_of_address": {
  "polite":      "中立·尊称/陌生",
  "friendly":    "中立·熟人/朋友",
  "intimate":    "中立·铁哥们/私下",
  "self":        "自称",
  "male_note":   "已知男时的特色称谓（可选）",
  "female_note": "已知女时的特色称谓（可选）"
}
```

### 5.3 gen 脚本渲染（片段最顶部，对齐 wen 代字 P1）

片段在「灵魂词」之前插入「称谓」段：

```
## 〇、称谓（先叫对人）
- 尊称/陌生：{polite}
- 熟人/朋友：{friendly}
- 铁哥们/私下：{intimate}
- 自称：{self}
{若有 *_note}已知男 → {male_note}；已知女 → {female_note}

勿预设性别，默认中立称谓；从对话线索判断后再换；用户曾明示或于 .claude/fangyan.local.md 固定则从之。浓度愈高用愈亲昵的称谓。
```

## 6. 组件三 · 工程里程碑庆祝（first-class·对齐 wen P4）

`元规则.md` 新增「## 工程里程碑庆祝」段：

```
## 工程里程碑庆祝

遇真正的工程里程碑（大 feature 落成、围因已久的 bug 修复、发版、长调试后
测试全绿等；AI 自识或用户明示），以当前方言区的民俗形式（东北二人转、
云南山歌、陕西秦腔、粤语饮茶舞狮、北京相声…）庆祝一两句，化用到编程语境
——取其庆祝味与符号，勿照搬人生仪式。

- 仅对话层：庆祝只在口头，绝不写进落盘之物。
- 勿凡回合皆庆：只对真里程碑，普通回合不庆。
- Emoji 点缀：可缀一个对应民俗符号的 Emoji（见下），不每句撒；正式/商务免。
- 素材：具体民俗形式与符号 → Read references/民间习俗.md §1/§2。
```

触发：**两者都认**（AI 自识 + 用户明示）。浓度/得体规则同样适用（铁哥们放开、正式收着）。

## 7. 组件四 · Emoji（方言符号映射·克制）

化解矛盾：语料 `do_not` 警告「勿太多 emoji」（东北爷们儿不爱发），但用户要 Emoji 传情绪。解法——**民俗符号映射 + 克制**：

从 `民间习俗.md §1`「关键民俗符号」取各方言 Emoji（草案，实施时按符号表定）：

| 方言 | 民俗符号 Emoji |
|---|---|
| 北京话 | 🏮 🎭（京剧/胡同） |
| 东北话 | 🍺 🎤（杀猪菜/二人转） |
| 云南话 | 🍄 🔥（菌子/火把节） |
| 陕西话 | 🍜 🥁（biangbiang 面/秦腔鼓） |
| 粤语 | 🍵 🦁（饮茶/舞狮） |
| 闽南话 | 🏮 🛕（拜拜/宗祠） |
| 成都/重庆 | 🌶️ 🍲（辣椒/火锅） |
| 上海话 | ☕ 🏘️（咖啡/弄堂） |
| …… | 按各方言符号补全 |

**用法**：庆祝/高光时缀 1 个，不每句撒；正式/商务场景免；**落盘之物零 Emoji**（硬约束）。

> Emoji 表暂以映射规则形式并入元规则/片段说明，不单列文件（YAGNI）。

## 8. 数据流

```
/fangyan <方言>
  └─ fangyan.sh
       ├─ cat 元规则.md        ← +「落盘不退化」「工程里程碑庆祝」
       └─ cat 方言注入/<方言>.md ← 顶部 +「称谓」段（自 terms_of_address 渲染）
                                    深入取阅 §七 已指向 民间习俗.md（里程碑素材）
模型按需 Read：
  民间习俗.md（民俗形式/符号/人生大事模板）｜ 方言字典.json（深度词汇）｜ 方言报告/<方言>.md
```

`方言灵魂.json` 是 gen 脚本唯一数据源；称谓表写入它，片段即自动渲染。

## 9. 测试

- **gen 测试**（`tests/test_gen.sh` 增断言）：13 片段均含「称谓」段；东北话片段默认称谓含**中立**「老铁」（非男定向的「大兄弟」），并含性别变体注记（大兄弟／大妹子）；各片段含 `.claude/fangyan.local.md` 固定机制说明。
- **元规则测试**（新建 `tests/test_meta.sh`）：`元规则.md` 含「落盘不退化」「术语白名单」「工程里程碑庆祝」关键字；含术语白名单样本（function/API/git/npm）。
- **handler 集成**：`bash commands-handlers/fangyan.sh 东北话` 输出含「称谓」「老铁」「落盘不退化」。
- **现有 3 套件**全过（gen/session-start/fangyan 不回归）。
- **运行时不退化**（人工验收，非自动化）：加载方言后要求写代码/commit，确认落盘物零方言味零 Emoji、术语原样、技术判断准确。
- **性别适配**（人工验收）：女性用户场景下，AI 默认用中立称谓、不冒称男性；用户透露或固定后正确切换。
